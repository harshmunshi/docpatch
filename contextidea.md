# DocPatch as a Context Catalogue

> **Brainstorm.** Not a spec. Ideas are marked by status: [works today], [needs building], [open question].

---

## The reframe

DocPatch today solves: *edit one node of a document, put it back losslessly.*

This document explores the adjacent question: *can DocPatch manage context itself — not just files, but any long-form content an LLM needs to reason over?*

The short answer is yes, and the reason is that the core value of DocPatch is not the parsing — it is the **stable, addressable, fingerprinted namespace over content**. Once a piece of text is in a DocTree, it has:

- a deterministic ID you can reference across time and re-parses
- a fingerprint that tells you the moment content changes
- a metadata slot for enrichment (scores, timestamps, provenance)
- a `walk()` interface for systematic traversal
- `content_skeleton()` for free summarisation without a model call
- lossless round-trip via `raw_span`

None of these properties care whether the source was a Markdown file, a JSON object, a conversation log, or a string generated by an LLM thirty seconds ago. **The format is incidental. The addressability is the point.**

---

## The three kinds of content, and where DocPatch stands today

| Kind | Examples | Current status |
|------|----------|----------------|
| **Structured** | Markdown, JSON, DOCX, HTML | Works today |
| **Semi-structured** | Logs, CSVs, JSONL, transcripts | Parseable with strategy — needs new parsers |
| **Flat** | Plain prose, raw LLM output, shell output | Needs chunking layer before DocTree |

The interesting design question is how to handle semi-structured and flat content without destroying the properties that make DocTree useful. The answer is: **impose the hierarchy, document the strategy used**.

## Part 1 — Making a DocTree out of anything

### The hierarchy is the strategy

Every file has at least one dimension of structure you can lift into a tree. The question is which dimension to choose as the primary axis, and how deep to go.

| Source | Natural hierarchy | DocTree primary axis |
|--------|-------------------|----------------------|
| Nginx log | timestamp → endpoint → status | status family → endpoint → entry |
| Python source | module → class → method | class → method → statement |
| CSV / JSONL | row → cell | schema column groups → row → cell |
| Transcript | session → speaker → turn | speaker → turn → sentence |
| Plain prose | none explicit | topic segment → paragraph → sentence |
| LLM conversation | time → role → content | role sequence → turn → paragraph |
| Shell session | command → output | command block → stdout/stderr |

The strategy choice is part of the DocTree's provenance — stored in `root.metadata["parse_strategy"]`.

### Five chunking strategies for flat content

**1. Delimiter-based** *(fast, zero ML)*
Split on blank lines, `---`, `\f` (form feed), or configurable regex. Produces a 2-level tree: root → CHUNK[]. Simple and deterministic; the default for plain text files.

```python
tree = parse_flat(text, strategy="delimiter", delimiter=r"\n\n+")
```

**2. Line-pattern** *(for logs, transcripts, code)*
Split when a line matches a pattern — a timestamp regex, a `def`/`class` keyword, a `Speaker:` pattern. Produces a 2 or 3-level tree depending on nesting depth.

```python
tree = parse_log(data, line_pattern=r"^\d{4}-\d{2}-\d{2}T")
tree = parse_transcript(data, speaker_pattern=r"^(\w[\w ]+):\s")
tree = parse_code(data, language="python")  # uses ast.parse internally
```

**3. Fixed-size token window** *(predictable cost, worst semantics)*
Chop at every N tokens. IDs are stable (`chunk:document:root/chunk#42`) but semantic boundaries are arbitrary. Useful when downstream tools impose a hard chunk size.

```python
tree = parse_flat(text, strategy="fixed", chunk_tokens=512)
```

**4. Semantic chunking** *(best quality, costs embeddings)*
Embed sentences; find cosine similarity drops; split where the topic shifts. The embedding model is provided by the caller — DocPatch does not bundle one. The strategy only runs once at parse time; after that the tree is static.

```python
tree = parse_flat(
    text,
    strategy="semantic",
    embed_fn=my_embed_function,  # (str) -> list[float]
    min_chunk_tokens=128,
    max_chunk_tokens=1024,
)
```

**5. LLM-guided** *(highest fidelity, expensive)*
Send the document to an LLM and ask it to identify natural section boundaries with short labels. Treats the parse step as a one-time index-build, not a real-time operation. Appropriate for content that will be accessed many times.

```python
tree = parse_flat(text, strategy="llm", model=my_client, max_sections=20)
```

### The CHUNK node type

All flat-to-tree strategies produce CHUNK nodes. CHUNK is the generic container for a segment of content with no native structural type.

Proposed addition to `NodeType`:

```python
class NodeType(StrEnum):
    # ... existing types ...

    # context / flat content
    CHUNK       = "chunk"      # generic flat-text segment
    TURN        = "turn"       # one speaker turn in a conversation
    LOG_ENTRY   = "log_entry"  # one log line with structured fields
    RECORD      = "record"     # one row in a tabular / JSONL source
    FRAME       = "frame"      # a time-bounded window (streaming / real-time)
    FUNCTION    = "function"   # a named function / method block (code)
    CLASS       = "class_def"  # a class definition (code)
```

A CHUNK node carries extra metadata set at parse time:

```python
{
    "chunk_index": 7,
    "token_count": 312,        # pre-computed; no re-scan needed
    "start_byte": 4096,        # position in source
    "end_byte": 5412,
    "parse_strategy": "semantic",
    "summary": None,           # populated if the chunk was summarised
    "relevance": None,         # set by a scoring pass
}
```

The existing `Node` model needs no changes — all of this lives in `metadata`.

### New `DocFormat` values

```python
class DocFormat(StrEnum):
    # ... existing ...
    PLAINTEXT  = "plaintext"
    LOG        = "log"
    TRANSCRIPT = "transcript"
    CSV        = "csv"
    JSONL      = "jsonl"
    CODE       = "code"       # language in metadata
    CONVERSATION = "conversation"  # OpenAI / Anthropic message list format
```

### What the 2-level flat tree enables

Even a degenerate tree `(root → CHUNK[])` is valuable because:

- The locator can search CHUNK content via `content_skeleton()` — no change to the locate pipeline
- Any chunk can be edited, summarised, or deleted using existing patcher operations
- Chunks have stable IDs — a vector database can index by `chunk.id` and the index survives re-parses (unchanged fingerprints = unchanged embeddings)
- `splice()` can insert new chunks (from a tool output) into the tree at any position
- Storage adapters preserve all chunks — you get history and diff for free in v0.5

## Part 2 — Context as a first-class thread

### The problem with the current model

Right now DocPatch has one entry point: a file on disk. The conversation that led to the edit, the tool outputs that informed it, the retrieved chunks that provided background — none of that is in the tree. When the session ends, that context is gone.

The proposal: **treat every distinct source of context as its own DocTree thread**, and give threads the same operations you give documents.

### ContextThread

```python
class ContextThread(BaseModel):
    id: str                    # deterministic from source
    source_kind: Literal[
        "file",                # a document on disk
        "conversation",        # an LLM message history
        "tool_output",         # stdout, API response, search result
        "retrieval",           # retrieved chunks from a vector search
        "generated",           # text produced by an LLM in this session
    ]
    tree: DocTree
    created_at: datetime
    updated_at: datetime
    token_count: int           # pre-computed; drives budget planning
    metadata: dict[str, Any]   # source path, model_id, query, etc.
```

A **session** is a bag of ContextThreads. Threads are independent trees. They can cite each other via node IDs using the `refs/` graph (planned v0.6, but the ID-stability makes this possible without any extra infrastructure today).

### What threads look like in practice

| Thread | Source kind | Root type | Children |
|--------|-------------|-----------|----------|
| `spec.md` | file | DOCUMENT | HEADING, PARAGRAPH, CODE_BLOCK, … |
| LLM conversation so far | conversation | DOCUMENT | TURN (role=user\|assistant\|tool) |
| `grep` output | tool_output | DOCUMENT | RECORD (one per match line) |
| Web search results | tool_output | DOCUMENT | RECORD (one per result) |
| Retrieved spec chunks | retrieval | DOCUMENT | CHUNK (from vector search) |
| LLM draft of new section | generated | DOCUMENT | HEADING, PARAGRAPH |

The caller decides what enters a session. DocPatch just provides the indexing.

### Cross-thread operations [needs building]

```python
class ContextCatalogue:
    threads: list[ContextThread]

    def find(self, query: str, budget_tokens: int) -> list[tuple[ContextThread, NodeRef]]:
        """Locate the nodes most relevant to a query, within a token budget."""

    def merge(self, source: ContextThread, target: ContextThread,
              instruction: str, model: ModelClient) -> DocTree:
        """Weave relevant parts of source into target using a patch instruction."""

    def diff(self, thread_a: ContextThread, thread_b: ContextThread) -> list[NodeRef]:
        """Return node IDs that differ between two trees of the same structure."""

    def cite(self, from_id: NodeRef, from_thread: str,
             to_id: NodeRef, to_thread: str, label: str) -> None:
        """Record that from_id was informed by to_id."""
```

## Part 3 — Context window planning

This is the highest-leverage feature the context catalogue enables.

### The problem

You have:
- A 100-page spec → 50K tokens
- A 200-turn conversation → 30K tokens
- 10 tool outputs → 8K tokens each → 80K tokens
- **Total: 160K tokens. LLM window: 128K tokens.**

Today you either truncate (lose information) or summarise the whole thing (one big lossy pass, expensive).

### The DocPatch approach

Because every node has a `token_count` in metadata and a relevance score, you can solve this as a **knapsack problem**:

```
Maximise relevance  ∑ relevance(node)
Subject to          ∑ token_count(node) ≤ budget
                    if node is selected, ancestors are selected (breadcrumb rule)
                    TURN nodes obey recency bias (last N turns always in)
```

```python
class ContextWindowPlanner:  # [needs building]

    def plan(
        self,
        catalogue: ContextCatalogue,
        query: str,
        budget_tokens: int,
        *,
        always_include_last_n_turns: int = 10,
    ) -> list[tuple[ContextThread, NodeRef]]:
        """
        Return the (thread, node_id) pairs that fill the budget
        with the highest relevance to query.
        
        Strategy:
        1. Always include all skeleton nodes (content_skeleton) — very cheap.
        2. Score remaining nodes: keyword overlap, recency, explicit weight.
        3. Greedily fill budget: highest score first, respecting the
           ancestor-inclusion constraint.
        4. Reserve a slot for the last N turns regardless of score.
        """
```

The result is a **composited context payload** — a sequence of node contents assembled from multiple threads, each with a header like `[spec.md § Rate Limiting]` or `[Turn 47 — user]`. The LLM gets exactly what it needs and nothing more.

### What this buys at scale

| Total context | Naive approach | DocPatch planned context |
|---------------|----------------|--------------------------|
| 10K tokens | Fits easily | Same |
| 50K tokens | Marginal | ~2-5K tokens (skeleton + relevant nodes) |
| 200K tokens | Must truncate | ~5-10K tokens — nothing lost, just not fetched yet |
| 1M tokens | Impossible | ~5-10K tokens — a document database |

The LLM's effective recall doesn't degrade with document length because it operates on a curated slice, not a dump.

## Part 4 — LLM conversation history as a DocTree

This is a specific, immediately buildable application of the above ideas.

### The problem with conversation history today

Conversation history is a dumb list of `{"role": "...", "content": "..."}` dicts. When it exceeds the context window, you truncate from the front or summarise everything — both lossy.

### Conversation as DocTree

```
DOCUMENT: conversation-abc123
  TURN: turn:…/user#0      {role: "user", turn_index: 0, tokens: 47}
    PARAGRAPH: "Can you review the rate-limiter spec?"
  TURN: turn:…/assistant#1  {role: "assistant", turn_index: 1, tokens: 312}
    PARAGRAPH: "Sure. Three issues stand out:"
    LIST:
      LIST_ITEM: "The token bucket doesn't handle burst correctly"
      LIST_ITEM: "No mention of Redis failover"
  TURN: turn:…/tool#2       {role: "tool", tool_name: "read_file", tokens: 4200}
    CODE_BLOCK: <contents of rate_limiter.py>
  TURN: turn:…/user#3       {role: "user", turn_index: 3, tokens: 18}
    PARAGRAPH: "Fix the burst issue in the spec."
```

Now you can:

```python
# Find the turn where burst was discussed
result = locator.locate(conv_tree, "the turn about token bucket burst")

# Summarise turns older than 100 to save tokens
old_turns = [n for n in conv_tree.walk()
             if n.type == NodeType.TURN and n.metadata["turn_index"] < 100]
patches = [summarizer.apply(conv_tree, t.id, "Summarize in one sentence", model)
           for t in old_turns]
compressed = splice(conv_tree, patches)
# compressed.token_count() → ~3K tokens instead of 30K
# original raw_spans still on disk — nothing lost

# Reconstruct the full conversation on demand
full = splice(conv_tree, [])   # no patches = emit raw_spans = original
```

### The Summarize operation [needs building]

A new `patcher/summarize.py` that differs from replace:
- Does **not** replace `content`
- Stores the summary in `metadata["summary"]`
- Sets `content` to the summary
- Leaves `raw_span` intact (original content is losslessly preserved)
- Marks `metadata["summarised"] = True`

An `expand()` operation reverses this by restoring `content` from `raw_span`.

This is the key semantic difference: `replace` changes the source of truth; `summarize` creates a compressed *view* while the original is preserved in `raw_span`.

## Part 5 — New parsers sketched

### `parsers/plaintext.py`

```python
def parse(data: bytes, *, strategy: str = "delimiter",
          embed_fn=None, model=None, chunk_tokens: int = 512) -> DocTree:
    """Parse plain text into a flat CHUNK tree."""

def serialize(tree: DocTree) -> bytes:
    """Reconstruct original bytes from CHUNK raw_spans."""
```

Round-trip invariant holds because each CHUNK stores `raw_span`. Concatenation of all CHUNK raw_spans = original file.

### `parsers/transcript.py`

Input:
```
Alice: Sure, let me check that.
Bob: The issue is on line 42.
Alice: Got it, fixing now.
```

Output:
```
DOCUMENT
  TURN {speaker: "Alice", turn_index: 0}
    PARAGRAPH: "Sure, let me check that."
  TURN {speaker: "Bob", turn_index: 1}
    PARAGRAPH: "The issue is on line 42."
  TURN {speaker: "Alice", turn_index: 2}
    PARAGRAPH: "Got it, fixing now."
```

Speaker names become groupable: `tree.speakers()` → `{"Alice", "Bob"}`.

### `parsers/log.py`

Input (nginx access log):
```
192.168.1.1 - - [01/Jan/2026:00:00:01 +0000] "GET /api/users" 200 1234
192.168.1.2 - - [01/Jan/2026:00:00:02 +0000] "POST /api/login" 401 56
```

Output with `group_by="status_family"`:
```
DOCUMENT
  SECTION {status_family: "2xx"}
    LOG_ENTRY {ip: "192.168.1.1", ts: ..., method: "GET", path: "/api/users", status: 200}
  SECTION {status_family: "4xx"}
    LOG_ENTRY {ip: "192.168.1.2", ts: ..., method: "POST", path: "/api/login", status: 401}
```

With `group_by="endpoint"` the same data organises differently. The grouping strategy is part of the parse-time metadata — a downstream patcher can ask "how was this tree structured?" from `root.metadata["group_by"]`.

### `parsers/code.py`

Uses `ast` (Python), `tree-sitter` (multi-language), or regex fallback. Produces a real 3-level tree:

```
DOCUMENT: server.py
  CLASS: RateLimiter  {lineno: 89, bases: ["object"]}
    FUNCTION: __init__   {lineno: 91, args: ["self", "capacity", "rate"]}
      CODE_BLOCK: <body>
    FUNCTION: check      {lineno: 102, args: ["self", "key"]}
      CODE_BLOCK: <body>
  FUNCTION: handle_request  {lineno: 42, args: ["req", "ctx"]}
    CODE_BLOCK: <body>
```

Now `edit_doc(tree, "Add retry logic to the check method")` sends only the `check` function body to the model — not the whole file.

### `parsers/conversation.py`

Input: OpenAI / Anthropic message list (JSON array):
```json
[
  {"role": "user", "content": "Review this spec."},
  {"role": "assistant", "content": "I see three issues..."}
]
```

Output: the TURN-based DocTree shown in Part 4. The parser handles `content` as a list of content blocks (for Anthropic tool use) or a plain string.

## Part 6 — Integration with vector search

Node IDs solve the "stale index" problem in vector databases.

```python
# Index once
tree = parse_flat(data)
for node in tree.walk():
    if node.type == NodeType.CHUNK and node.content:
        vector_db.upsert(
            id=node.id,                    # stable across re-parses
            vector=embed(node.content),
            payload={"fingerprint": node.fingerprint, "source": path}
        )

# On re-parse: only re-embed nodes whose fingerprint changed
new_tree = parse_flat(updated_data)
for node in new_tree.walk():
    old = vector_db.fetch(node.id)
    if old and old["fingerprint"] == node.fingerprint:
        continue   # skip — content unchanged, embedding still valid
    vector_db.upsert(id=node.id, vector=embed(node.content), ...)

# Retrieve and patch in one shot
results = vector_db.query(query_vector, limit=5)
# results carry node IDs → skip the locator entirely
node_id = results[0]["id"]
patch = patcher.apply(tree, node_id, "Expand this section with an example", model)
```

The fingerprint ensures embedding freshness. The stable ID ensures the vector index and the DocTree stay in sync without a full re-index on every file change.

## Part 7 — Open questions

### Q1: Who owns the parse strategy for flat content?

For structured files (Markdown, JSON), the format determines the parse strategy. For flat files, the caller does. Should the strategy be:
- a parameter passed at parse time (explicit, per-call)
- a config in `Settings` (global default)
- an auto-detected field (infer from content)

Recommendation: parameter at parse time, with `Settings` providing a default. Auto-detection is too error-prone to be reliable.

### Q2: Is `CHUNK` too generic?

`CHUNK` is a catch-all for "a flat text segment with no native structural type". This risks being a grab-bag. The alternative is to require callers to pick a more specific type. But `LOG_ENTRY`, `TURN`, `RECORD` are all specialisations of the same concept — a unit of flat content with metadata. The `metadata["parse_strategy"]` field disambiguates if needed.

Recommendation: keep CHUNK as the base. The specific types (`TURN`, `LOG_ENTRY`, etc.) are just CHUNKs with a known metadata schema. The NodeType enum distinguishes them.

### Q3: Does the locator work on CHUNK trees?

`SemanticLocator` uses `content_skeleton()`. Currently `content_skeleton()` surfaces headings, code blocks, and tables. It needs to also surface CHUNKs by their first sentence or `metadata["summary"]` if present.

Change required: one `elif node.type == NodeType.CHUNK` branch in `content_skeleton()`. [Small, buildable now.]

### Q4: How do threads share a namespace?

If thread A and thread B both parse the same markdown file, their root IDs will be identical (`document:root#0`). Cross-thread node references need the thread ID as a prefix: `{thread_id}::{node_id}`.

This is fine as a convention but needs to be encoded in `RefGraph` (v0.6). For now, callers using cross-thread citations should prefix manually.

### Q5: Should conversation threads be editable?

The `summarize` operation makes old TURN nodes smaller. But should you be allowed to `replace` a TURN? (i.e., rewrite what the assistant said.) This breaks the audit invariant. Recommendation: restrict TURN patching to `summarize` and `annotate` only; `replace` on a TURN raises a `PolicyError`.

### Q6: How does `serialize` work for multi-thread sessions?

Each thread serializes independently. A session is not a single document — it does not have a single serializer. The ContextCatalogue has an `export()` method that produces a structured bundle (similar to the `.dpx/` sidecar format but containing multiple trees).

## Summary: what to build, in what order

| Feature | Module | Effort | Enables |
|---------|--------|--------|---------|
| `parse_flat()` with delimiter strategy | `parsers/plaintext.py` | Small | Everything below |
| `CHUNK`, `TURN`, `LOG_ENTRY`, `RECORD` node types | `core/types.py` | Trivial | New parsers |
| `content_skeleton()` CHUNK support | `core/tree.py` | Trivial | Locating CHUNKs |
| `parse_transcript()` | `parsers/transcript.py` | Small | Conversation indexing |
| `parse_conversation()` | `parsers/conversation.py` | Small | LLM history management |
| `patcher/summarize.py` | `patcher/` | Medium | Compression |
| `parsers/log.py` | `parsers/` | Medium | Log analysis |
| `parsers/code.py` (ast-backed) | `parsers/` | Medium-large | Code editing |
| `ContextThread`, `ContextCatalogue` | `context/` | Large | Multi-source sessions |
| `ContextWindowPlanner` | `context/` | Large | Budget-aware context |
| Cross-thread `RefGraph` | `refs/` | Large | Citation tracking |

The first four items (`parse_flat`, node types, `content_skeleton` CHUNK support, `parse_transcript`) are mechanical extensions of existing code — no new architectural layers, no new dependencies. They are the foundation everything else sits on.

The `ContextWindowPlanner` is the most novel piece. It is also the one that has no analogue in the existing ecosystem — every current approach either truncates or dumps. A planner that can surgically assemble a context payload from an addressed tree is a genuinely new primitive.
